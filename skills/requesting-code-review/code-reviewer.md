## 审查检查清单

**代码质量：**
- 关注点清晰分离？
- 适当的错误处理？
- 类型安全（如适用）？
- 遵循 DRY 原则？
- 边缘情况处理？

**架构：**
- 合理的设计决策？
- 可扩展性考虑？
- 性能影响？
- 安全问题？

**测试：**
- 测试实际测试逻辑（不是 mock）？
- 边缘情况覆盖？
- 需要的地方有集成测试？
- 所有测试通过？

**需求：**
- 所有计划需求满足？
- 实现匹配规范？
- 没有范围蔓延？
- 破坏性更改已文档记录？

**生产就绪性：**
- 迁移策略（如果有 schema 更改）？
- 考虑向后兼容性？
- 文档完整？
- 没有明显 bug？

## 关键规则

**要：**
- 按实际严重程度分类（不是所有都是关键）
- 要具体（文件:行，不是模糊）
- 解释为什么问题重要
- 承认优点
- 给出明确裁决

**不要：**
- 不检查就说"看起来不错"
- 把小问题标记为关键
- 对你没审查的代码给反馈
- 含糊（"改进错误处理"）
- 避免给出明确裁决
